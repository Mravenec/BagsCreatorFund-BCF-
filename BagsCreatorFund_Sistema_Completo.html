BAGS HACKATHON 2025

BagsCreatorFund
Sistema Completo

Un protocolo de rondas de financiamiento on-chain sobre Solana donde los creadores lanzan tokens en Bags, abren campañas de funding, permiten que cualquier persona — con wallet Web3 o desde cualquier exchange (CEX) — compre posiciones, y usan los fondos acumulados para reinvertir en su propio token, generando presión de compra orgánica.

01 — Visión general

¿Qué hace BCF?

BCF conecta tres capas en un solo flujo económico coherente:

🚀

Lanzamiento de token

El creador genera su token directamente en Bags Mainnet (Meteora DBC) desde la interfaz, con fee-share configurado a su favor.

🎯

Rondas de financiamiento

Por cada token puede abrir campañas ilimitadas donde participantes compran posiciones numeradas (00–99) con SOL real.

🔥

Flywheel de reinversión

Los fondos acumulados en tesorería se usan para comprar el propio token en Bags, creando presión de compra orgánica y sostenida.

🏦

CEX sin fricción

Usuarios de Binance, Coinbase, OKX, Kraken o cualquier exchange participan enviando SOL a una dirección vault única por posición — sin memo, sin wallet Web3 obligatoria.

El problema que resuelve: Los tokens lanzados en Bags no tienen mecanismo de presión de compra sostenida después del lanzamiento inicial. BCF crea ese motor: cada ronda de financiamiento sin ganador se convierte automáticamente en una orden de compra programada del token.

Cinco pantallas, un flujo completo

03 — Creación de token

Lanzar un token en Bags desde BCF

1

Project Info

El creador ingresa nombre, símbolo, descripción, imagen e imagen (URL). BCF llama a POST /token-launch/create-token-info (FormData) en la API de Bags.

2

Launch Type — LAUNCH TYPE dropdown

El creador elige uno de los 4 modos de fee disponibles en la API (bagsConfigType):
• Founder Mode — 2% flat, earn 1% of total trading volume (UUID: fa29606e…)
• Low Early → High After — 0.25% pre-migration → 1% post (UUID: d16d3585…)
• High Early → Low After — 1% pre-migration → 0.25% post (UUID: a7c8e1f2…)
• Paper Hand Tax Mode — 10% flat, 50% added to LP (UUID: 48e26d2f…)
Company Mode está disponible post-lanzamiento desde el Dashboard (flujo /incorporate/*).

3

Fee Sharing — hasta 100 claimers

BCF configura claimersArray y basisPointsArray (suma = 10 000 bp = 100%). Por defecto el creador recibe el 100%, pero puede distribuir fees entre múltiples wallets, apps, o colaboradores. Editable después del lanzamiento desde el Dashboard via POST /fee-share/admin/update-config.

4

Ownership — Initial Buy (opcional)

El creador puede comprar tokens antes que nadie (buy at launch). Se especifica el monto en SOL (initialBuyLamports). La UI muestra presets: Skip / 0.1 / 0.5 / 1 / 2.5 SOL.

5

Review & Launch — firma Mainnet

El usuario revisa todo y firma con Phantom/Solflare. BCF ejecuta los 4 pasos de la API de Bags: metadata → fee-share config TX → launch TX → broadcast. El token queda vivo en bags.fm/token/<mint>.

Fallback DevNet: Si la API de Bags no responde, el sistema usa un mint simulado y continúa creando el ProjectAccount on-chain. Todas las funciones de campaña funcionan normalmente para testing.
04 — Arquitectura on-chain

Jerarquía de cuentas (PDAs)

Tesorería real: treasury_lamports en cada ProjectAccount siempre coincide con el saldo real en lamports de esa cuenta PDA. No es un contador contable — es dinero físico dentro de esa dirección on-chain.
05 — Ciclo de vida de una campaña

Tres estados, dos posibles destinos

Aleatoriedad sin oracle: resolve_campaign lee el SlotHashes sysvar de Solana — un hash del slot actual de la red. El resultado es determinístico, público, reproducible por cualquier nodo, y no depende de Chainlink ni ningún servicio externo.

El rango siempre es 00–99, incluyendo posiciones no vendidas. Eso introduce la posibilidad real de que nadie gane, lo que alimenta la tesorería del creador.

Camino 2 — Exchange / CEX (Binance · Coinbase · OKX · Kraken · cualquier dirección Solana)

Sin wallet Web3 obligatoria, sin memo. Cada posición tiene su propio vault PDA. El watcher crea el vault on-chain en cuanto el usuario ingresa su dirección, y asigna la posición automáticamente cuando el SOL llega.

Watcher — arranca automáticamente con bash scripts/deploy.sh (DevNet) o bash scripts/deploy-mainnet.sh (Mainnet).
Para desarrollo sin re-deploy: npm run dev:full (Vite + watcher en paralelo).

El deploy.sh espera hasta 15 s a que el endpoint /health responda antes de confirmar que el watcher está listo. Si el proceso falla, muestra las últimas líneas de watcher.log para diagnóstico.
Fallback automático: Si el SOL del exchange llegó al vault antes de que el watcher lo inicializara, el contrato expone recordExternalPayment — asigna la posición sin vault. El pago queda registrado en blockchain en cualquier caso.
Sobre el monto a enviar: Los exchanges cobran un fee de retiro propio (generalmente 0.001–0.02 SOL) que se descuenta del saldo del usuario, no del monto enviado. Para garantizar que el vault reciba el precio completo, la UI muestra un monto recomendado = precio + buffer de 0.01 SOL. Cualquier exceso por encima del precio va al sweeper como tip.
07 — Watcher automático

HTTP API + dos fases de polling

El watcher corre como proceso Node.js separado (node scripts/watcher.mjs) con su propio keypair Solana. El servidor HTTP arranca primero (antes de que Anchor inicialice) — el puerto 3001 siempre está disponible incluso durante la carga inicial.

4 endpoints HTTP:
GET  /health — estado del servicio; responde siempre, incluso durante init. Muestra si Anchor está listo.
POST /watch-vault — pre-crea el vault on-chain en cuanto el usuario ingresa su dirección Solana. Así el vault existe antes de que llegue el SOL.
POST /sweep-now — asigna la posición CEX. El frontend lo llama automáticamente al detectar el pago, y el botón "🎯 Take My Position" sirve de fallback manual.
POST /refund-vault — reembolsa un vault cuya posición ya fue tomada por otra persona. El frontend lo llama inmediatamente al detectar el conflicto; el poll loop también lo llama de forma proactiva.
FaseDescripciónFrecuencia
① Sweep Escanea todos los PositionVault on-chain con balance ≥ precio → sweepPositionVault → posición asignada, vault cerrado. Si la TX falla con PositionTaken, activa auto-reembolso de emergencia. Cada 15s
① b. Reembolso Anti race-condition: Escanea todos los vaults activos. Si la posición del vault ya fue asignada a otra dirección (por un usuario wallet o un sweep anterior), llama refundPositionVault → todo el SOL regresa al recipient automáticamente. Cada 15s
② Settlement Detecta campañas en estado Settled. Con ganador: push_prize → SOL al winner. Sin ganador: route_no_winner_to_treasury → SOL al proyecto. Cada 15s
🛡️ Protección contra conflictos de posición (race condition):

Si dos personas intentan comprar la misma posición al mismo tiempo:
• Usuarios con wallet: el contrato Anchor rechaza la segunda transacción con PositionTaken antes de transferir SOL — el dinero nunca sale.
• Usuarios CEX/vault: cada uno tiene un vault PDA único. El segundo sweep falla con PositionTaken. El watcher detecta esto y llama refundPositionVault automáticamente — el SOL regresa al recipient (menos el fee de red ~0.000005 SOL). El frontend también solicita el reembolso de forma inmediata y muestra una pantalla informativa.

Fallback automático — recordExternalPayment: Si el SOL del exchange llegó al vault PDA antes de que el watcher lo inicializara, el watcher usa recordExternalPayment. Asigna la posición directamente sin necesitar el vault. El pago queda registrado en blockchain en cualquier caso.
Premio CEX automático: Si la posición ganadora la compró un usuario CEX, el watcher detecta el estado Settled y llama push_prize automáticamente — SOL directo al recipient. Sin acción, sin wallet.
💰 Modelo económico del sweeper — auto-sostenido:
+ Vault rent reclamado ~0.002 SOL por sweep (el contrato cierra el vault y devuelve el rent al sweeper) + Tip del usuario Exceso sobre el precio de posición (ej: usuario envía 0.21, precio 0.20 → 0.01 SOL al sweeper) − Fee TX ~0.000005 SOL por transacción Solana

Con 0.05 SOL iniciales el sweeper puede ejecutar ~10,000 operaciones. El watcher expone un endpoint GET /health que muestra el accounting en tiempo real: sweeps exitosos, fees gastados, rent reclamado, tips acumulados y estimación de rentabilidad neta. El watcher pausa automáticamente si el balance baja de 0.005 SOL para no perder fees en reintentos.

08 — Entrega del premio

¿Cómo le llega el premio a un usuario CEX?

1

Posición asignada con recipient

Al hacer sweep, el contrato registra pos.owner = recipientAddr on-chain. Esta es la dirección que el usuario ingresó en la UI.

2

Campaña resuelve (cualquiera puede llamar)

Al vencer el deadline, cualquier usuario (o el watcher) llama resolve_campaign. Se lee el SlotHashes sysvar y se determina la posición ganadora.

3

Watcher detecta ganador · push_prize automático

El watcher escanea campañas en estado Settled. Si pos.filled == 1, llama push_prize. El contrato transfiere prize + total_collected SOL directamente a la wallet ganadora.

4

Sin acción requerida del ganador

El SOL llega automáticamente a la dirección recipientAddr. El usuario puede verificarlo en cualquier explorador de Solana. La UI también guarda localmente el historial de posiciones CEX.

Hint local + verificación on-chain: La UI guarda en localStorage el recipientAddr que el usuario ingresó al comprar por CEX (solo como hint de dirección). Cuando la campaña se resuelve, el banner de ganador se muestra únicamente si campaign.positions[winningPosition].owner on-chain coincide exactamente con ese recipientAddr.

Modificar localStorage no tiene efecto de seguridad: el campo owner vive en el contrato Anchor y solo puede ser escrito por sweepPositionVault o recordExternalPayment. El blockchain es la única fuente de verdad.
09 — Seeds de PDAs

Derivación determinística de todas las cuentas

// 1. Creator Registry — uno por wallet creadora seeds = ["registry", creator_wallet] // 2. Project Account — uno por token lanzado seeds = ["project", creator_wallet, project_index_as_u64_le8] // 3. Campaign Account — incluye project_index → sin colisión entre tokens seeds = ["campaign", creator_wallet, project_index_le8, campaign_count_le8] // 4. Position Vault — único por (campaña × posición × usuario CEX) seeds = ["vault", campaign_key, position_index_byte, recipient_key]
¿Por qué recipient en el vault seed? La dirección del vault es única para cada combinación de campaña, posición y usuario. Un vault para la posición #47 de la campaña A con recipient X no colisiona con el vault de la misma posición con recipient Y. Esto garantiza que cada usuario CEX tiene su propio escrow aislado.
10 — Redes

DevNet y Mainnet — mismo contrato

El contrato Anchor/Rust es el mismo código en ambas redes. La diferencia es únicamente el cluster y si el SOL es real.

Arquitectura de dos capas simultáneas: BCF siempre usa dos redes al mismo tiempo, sin importar el valor de VITE_NETWORK.

Capa 1 — Bags (siempre Mainnet): El token, el fee-share config, el trading, el claim de fees y el swap del flywheel van siempre a api.mainnet-beta.solana.com. Esto está hardcodeado en bags.js con mainnetConnection y no cambia.

Capa 2 — BCF Program (controlado por VITE_NETWORK): Los ProjectAccount, CampaignAccount, PositionVault, treasury y resolución van al cluster configurado — DevNet para desarrollo, Mainnet para producción.

Para producción completa solo se necesita: desplegar el programa Anchor en Mainnet, actualizar VITE_BCF_PROGRAM_ID_MAINNET y cambiar VITE_NETWORK=mainnet.

11 — Integración Bags

SDK de Bags — 4 recursos integrados

🚀

Token Launch

Flujo completo 4 pasos: create-token-info → fee-share/config → create-launch-transaction → firma Mainnet.

💸

Fee Share V2

claimersArray: [creator_wallet] — por defecto 100% de los fees de trading van al creador. Configurable en el lanzamiento con hasta 100 receptores.

Dashboard — Gestión post-lanzamiento:
• Manage Fee Sharing: actualiza claimers/% via POST /fee-share/admin/update-config (firma TX)
• Transfer Admin: transfiere la autoridad admin a otra wallet via POST /fee-share/admin/transfer-tx
• Claim Bags Fees: reclama fees acumulados via POST /fee-claiming/claim-transactions (v3)
• Lifetime Fees: muestra total histórico via GET /analytics/lifetime-fees
• Company Mode: inicia incorporación legal via endpoints /incorporate/*
🔄

Trade API

GET /trade/quote + POST /trade/swap para el flywheel. SOL de la tesorería → token del creador en Meteora.

🏦

Claim Fees

POST /token-launch/fee-claiming/claim-transactions (v3) para reclamar ingresos de trading acumulados directamente desde el Dashboard.
