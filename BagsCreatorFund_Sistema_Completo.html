BAGS HACKATHON 2025

BagsCreatorFund
Sistema Completo

Protocolo de rondas de financiamiento on-chain sobre Solana donde los creadores lanzan tokens en Bags, abren campañas de funding, y cualquier persona — con wallet Web3 o desde cualquier exchange CEX — compra posiciones numeradas. Los fondos acumulados se reinvierten en el propio token del creador, generando presión de compra orgánica y sostenida.

01 — Visión general

¿Qué hace BCF?

BCF conecta tres capas en un solo flujo económico coherente:

🚀

Lanzamiento de token

El creador genera su token directamente en Bags Mainnet (Meteora DBC) desde la interfaz, con fee-share configurable. El modo de fee se elige en el lanzamiento; los claimers son editables post-lanzamiento.

🎯

Rondas de financiamiento

Por cada token puede abrir campañas ilimitadas con 100 posiciones numeradas (00–99). Cada posición vale X SOL y distribuye una cantidad fija de tokens al comprador.

🔥

Flywheel de reinversión

Los fondos de la tesorería se usan para comprar el propio token en Bags via Trade API, creando presión de compra orgánica. El creador también puede recargar la tesorería directamente desde su wallet.

🏦

CEX sin fricción

Usuarios de Binance, Coinbase, OKX o cualquier exchange participan enviando SOL a una Burner Wallet temporal generada en el browser — sin memo, sin wallet Web3, sin intermediario backend.

El problema que resuelve: Los tokens lanzados en Bags no tienen mecanismo de presión de compra sostenida. BCF crea ese motor: cada ronda de financiamiento sin ganador alimenta la tesorería del creador, que se usa para recomprar el token automáticamente.
02 — Interfaz de usuario

Seis rutas, un flujo completo

🏠 Home — /

Muestra campañas activas en tiempo real, estadísticas globales (campañas totales, SOL en juego) y el estado en vivo de la conexión con la API de Bags. Botones de acceso rápido a Explore y New Token.

🔍 Explore — /explore

Catálogo completo de campañas con filtros por estado (activo, pendiente, resuelto) y categoría. Búsqueda por texto. Ordenado automáticamente: activos primero, luego pendientes, luego resueltos.

🪙 New Token — /create-token

Flujo guiado de 5 pasos: Project Info → Launch Type → Fee Sharing → Ownership (initial buy) → Review & Launch. Lanza en Bags Mainnet con fallback a simulación DevNet si la API falla.

📊 Dashboard — /dashboard

Panel del creador: tokens, campañas, tesorería, datos de mercado en vivo. Acciones: Withdraw, ⬇ Fund Treasury, 🔥 Reinvest, ⚙️ Fee Sharing, 💰 Claim Bags Fees. Solo accesible con wallet conectada.

➕ New Campaign — /create-campaign

Formulario de 3 pasos (Campaign Info → Funding Economics → Review). Se vincula a un token/proyecto existente del creador. Accesible desde el Dashboard o con ?project=N en la URL.

📋 Campaign Page — /campaign/:pda

Grid de 100 posiciones (00–99), panel de estadísticas en tiempo real, modal de participación con modo Wallet o modo CEX (Burner Wallet), y banner de ganador verificado on-chain tras el sorteo.

Wallets soportadas: Phantom · Solflare · Coinbase Wallet · Trust Wallet. Configuradas en App.jsx via @solana/wallet-adapter-*. El banner de red en la parte superior cambia entre DevNet (morado) y Mainnet (verde).
03 — Creación de token

Lanzar un token en Bags desde BCF — 5 pasos

1

Project Info

Nombre, símbolo (max 10 chars), descripción (min 20 chars), imagen URL, Twitter, website y categoría. BCF llama a POST /token-launch/create-token-info (FormData, no JSON) → devuelve tokenMint y metadataUri.

2

Launch Type — Fee Mode

Elige uno de los 4 modos de fee (bagsConfigType UUID):
• Founder Mode — 2% flat, creator recibe 1% del volumen total (UUID: fa29606e…) recommended
• Low Early → High After — 0.25% pre-migration → 1% post (UUID: d16d3585…)
• High Early → Low After — 1% pre-migration → 0.25% post (UUID: a7c8e1f2…)
• Paper Hand Tax Mode — 10% flat, 50% añadido a LP (UUID: 48e26d2f…)

3

Fee Sharing — hasta 100 claimers

La UI permite configurar claimersArray y basisPointsArray (suma = 10 000 bp). Importante: durante el lanzamiento, Bags API recibe siempre [creator, 100%] — es la única configuración válida en el momento de creación. Los claimers personalizados se aplican post-lanzamiento desde el Dashboard via POST /fee-share/admin/update-config.

4

Ownership — Initial Buy (opcional)

El creador puede comprar tokens antes que nadie. La UI ofrece presets: Skip / 0.1 / 0.5 / 1 / 2.5 SOL. Se pasa como initialBuyLamports al step 3 de la API.

5

Review & Launch — firma Mainnet

El usuario revisa todo y firma con su wallet. BCF ejecuta los 4 pasos de Bags API en secuencia: create-token-info → fee-share/config (+ firma si needsCreation=true) → create-launch-transaction → broadcast a Mainnet. Si algo falla, cae a modo simulación DevNet automáticamente y el proyecto BCF on-chain se crea igualmente. El token queda vivo en bags.fm/token/<mint>.

Company Mode (post-lanzamiento): Visible en la UI del token launch como opción adicional. Se inicia desde el Dashboard via POST /incorporate/payment → POST /incorporate → POST /incorporate/start. No forma parte del flujo de lanzamiento inicial.
04 — Arquitectura on-chain

Tres tipos de cuentas PDA

Tesorería real: treasury_lamports en cada ProjectAccount siempre coincide con el SOL físico en esa PDA. Aumenta cuando una campaña sin ganador transfiere su fondo, o cuando el creador deposita directamente con deposit_to_treasury. Disminuye con withdraw_treasury.
05 — Ciclo de vida de una campaña

Tres estados, dos posibles destinos

Aleatoriedad sin oracle: resolve_campaign lee el SlotHashes sysvar de Solana. El resultado es determinístico, público y reproducible. Cualquier usuario (no solo el creador) puede llamar esta instrucción una vez pasado el deadline — hay un botón visible en la Campaign Page para todos.

El rango siempre es 00–99, incluyendo posiciones no vendidas. Eso introduce la posibilidad real de que nadie gane, lo que alimenta la tesorería del creador.

06 — Formas de participar

Dos caminos de entrada

🔐 Modo Wallet — Web3 directa

El usuario conecta Phantom, Solflare, Coinbase Wallet o Trust Wallet. Hace clic en una posición disponible en el grid → firma la transacción → buy_position se ejecuta on-chain inmediatamente. Si la posición fue tomada entre el clic y la firma, el contrato Anchor rechaza con PositionTaken antes de mover fondos.

🏦 Modo CEX — Burner Wallet

El usuario ingresa su dirección Solana (recipient). El browser genera un Burner Wallet temporal único para ese (campaña × posición). El usuario envía SOL desde su exchange a esa dirección. El browser detecta el pago y firma buy_position automáticamente — cero backend, cero servidor.

Flujo detallado — Modo CEX (Burner Wallet)

Serverless — $0 Infraestructura: El Burner Wallet se genera con Keypair.generate() en el browser. El polling usa connection.getSignaturesForAddress + getParsedTransaction directamente contra el RPC. No hay ningún servidor, watcher ni proceso backend involucrado en el flujo CEX.
Reembolsos y Cambio — Tres Casos:
① Pago insuficiente: El UI muestra el déficit exacto ("Received 0.0190 SOL — need 0.0020 more") y continúa esperando — nunca hace nada prematuro.
② Cambio justificado (>0.00001 SOL / 10 000 lamports): Después de comprar la posición, el Burner devuelve el sobrante al recipient con un SystemProgram.transfer. El usuario recibe un toast de confirmación.
③ Dust (≤0.00001 SOL): No se genera TX de reembolso adicional — el costo de la propia TX superaría el monto. El dust queda en el burner y se consume en fees.
07 — Arquitectura serverless

Zero-Touch — Sin VPS, sin watcher, sin backend

Toda la lógica CEX vive en el browser del usuario. El frontend lee el blockchain directamente a través del RPC configurado, sin intermediarios centralizados.

FaseImplementación realGarantía de seguridad
① Generación del Burner Keypair.generate() en el browser via @solana/web3.js. La clave privada se guarda en localStorage con clave bcf_burner_{campaignPDA}_{positionIndex} para resistir refrescos. Se elimina automáticamente al completar la compra o el reembolso. Efímero. La clave privada solo existe mientras dura la transacción. Scope: una campaña × una posición.
② Detección de pago getSignaturesForAddress + getParsedTransaction cada 5 segundos. Se suman todos los transfer recibidos. Si el total ≥ precio + 0.0010 SOL (mínimo gas) → procede. Si es parcial → notifica déficit y espera. Si hay pago pero la posición ya fue tomada → ejecuta reembolso. Safe Buffer: Se solicita 0.0100 SOL extra para cubrir cualquier comisión de exchange y asegurar gas para el Browser Crank auto-payout.
③ Ejecución on-chain El Burner Wallet es el buyer que firma buy_position. El parámetro recipient se pasa como la dirección que el usuario ingresó. El contrato Anchor asigna pos.owner = recipient on-chain. El Burner wallet no puede ganar el premio: la posición pertenece al recipient on-chain. El Burner solo firma la compra.
④ Devolución de cambio Tras la compra, se lee el balance del Burner. Si supera 10 000 lamports, se ejecuta un SystemProgram.transfer del sobrante al recipient. Si es dust (≤10 000 lamports), no se genera TX adicional. El recipient siempre recibe el cambio. El dust mínimo queda en el Burner (valor inferior al costo de la TX de reembolso).
🛡️ Race condition — dos usuarios, misma posición:

• Wallet directa: el contrato Anchor rechaza la segunda transacción con error PositionTaken antes de transferir SOL — el dinero nunca sale de la wallet del usuario.
• CEX / Burner: La instrucción buy_position también falla con PositionTaken. El frontend detecta este error y ejecuta executeRefund inmediatamente, devolviendo todo el SOL al recipient. El usuario ve el panel de reembolso con un botón para elegir otra posición.

08 — Entrega del premio

¿Cómo llega el premio al ganador?

1

Posición asignada con recipient

Al comprar una posición CEX, buy_position registra pos.owner = recipientAddr on-chain. La Campaign Page guarda en localStorage el recipientAddr como hint local para mostrar el banner de ganador. El hint no tiene valor de seguridad — la fuente de verdad es siempre el blockchain.

2

Campaña resuelve — cualquiera puede llamar

Pasado el deadline, aparece el botón "🎲 Resolve Round Now" en la Campaign Page para cualquier usuario. Llama resolve_campaign: lee SlotHashes sysvar de Solana y determina la posición ganadora de forma determinística. No requiere el creador ni ningún operador.

3a

Ganador con wallet conectada → claim_prize

Si la posición ganadora fue comprada con wallet Web3, el ganador ve el botón "⚡ Claim Prize" tanto en la Campaign Page como en su Dashboard. Firma claim_prize y recibe prize_lamports + total_collected directamente en su wallet. Requiere que el ganador firme la TX.

3b

Ganador CEX → push_prize (Automático)

Si la posición fue comprada vía CEX, el sistema usa el Browser Crank. El Burner Wallet retiene 0.005 SOL de gas. En cuanto el ganador (o cualquier usuario) visita la web tras el deadline, el navegador detecta el premio y llama a push_prize automáticamente. El ganador recibe el premio sin haber tenido que conectar una wallet ni firmar nada.

4

Sin ganador → tesorería del creador

Si la posición ganadora estaba vacía, aparece el botón "🏦 Move Funds to Treasury" para el creador en la Campaign Page y el Dashboard. Llama route_no_winner_to_treasury: transfiere prize + collected al ProjectAccount PDA y actualiza treasury_lamports on-chain.

Verificación on-chain del ganador CEX: La Campaign Page consulta campaign.positions[winningPosition].owner directamente del blockchain. Solo muestra el banner "You won via Exchange payment!" si ese valor coincide exactamente con el recipientAddr guardado en localStorage. Modificar localStorage no activa el banner porque la condición real viene del contrato — el campo owner solo puede escribirse via buy_position o push_prize.
09 — Seeds de PDAs

Tres cuentas PDA — derivación determinística

// 1. Creator Registry — uno por wallet creadora · registra conteo de proyectos seeds = ["registry", creator_wallet] // 2. Project Account — uno por token lanzado · almacena treasury_lamports seeds = ["project", creator_wallet, project_index_as_u64_le8] // 3. Campaign Account (zero_copy) — una por ronda de funding · almacena 100 posiciones seeds = ["campaign", creator_wallet, project_index_le8, campaign_count_le8]
No existe PositionVault PDA. El flujo CEX usa un Burner Wallet (keypair efímero generado en el browser), no una cuenta PDA on-chain. El SOL enviado al Burner se usa directamente para pagar buy_position y el burner firma como buyer. No hay ninguna cuenta PDA de tipo "vault" por posición en el contrato actual.
CampaignAccount = zero_copy (4 918 bytes): Supera el límite BPF de stack de 4 096 bytes. Se usa #[account(zero_copy(unsafe))] + AccountLoader — la cuenta se mapea directamente desde la memoria de la cuenta, nunca se copia al stack. Las positions: [PositionInfo; 100] (4 000 bytes) son la causa del tamaño.
09.5 — Instrucción nueva — v0.2.1

deposit_to_treasury — Recarga directa desde wallet Web3

El creador puede añadir SOL a la tesorería de un proyecto directamente desde su wallet conectada, sin esperar a que una campaña resuelva.

En el Dashboard: Cada tarjeta de token muestra el botón ⬇ Fund Treasury from Wallet. Al hacer clic, un modal muestra el balance actual de la tesorería vs. el balance de la wallet, con presets rápidos (0.1 / 0.5 / 1 / 2 SOL). Al confirmar, se firma la instrucción depositToTreasury on-chain. Si la TX falla, no hay cambio de estado. 11 — Redes

DevNet y Mainnet — mismo contrato

El mismo binario Anchor/Rust se despliega en ambas redes. La diferencia es solo el cluster y si el SOL es real.

Arquitectura de dos capas simultáneas: BCF siempre usa dos redes al mismo tiempo, sin importar el valor de VITE_NETWORK.

Capa 1 — Bags (siempre Mainnet): El token, fee-share, trading y claim de fees van siempre a api.mainnet-beta.solana.com. Hardcodeado en bags.js como mainnetConnection.

Capa 2 — BCF Program (controlado por VITE_NETWORK): Los ProjectAccount, CampaignAccount, treasury y resolución van al cluster configurado — DevNet para desarrollo, Mainnet para producción.

12 — Integración Bags

SDK de Bags — 7 grupos de endpoints

🚀

Token Launch

4 pasos: create-token-info (FormData) → fee-share/config → create-launch-transaction → firma Mainnet. Fallback automático a mint simulado si la API no responde.

💸

Fee Share Admin

Post-lanzamiento: GET /fee-share/admin/list · POST /fee-share/admin/update-config (cambia claimers/%) · POST /fee-share/admin/transfer-tx (transfiere autoridad).

💰

Claim Fees

POST /fee-claiming/claim-transactions (v3) para reclamar fees de trading acumulados. Se firman individualmente desde el Dashboard. GET /fee-claiming/claimable-positions para ver cuánto hay acumulado.

🔄

Trade API — Flywheel

GET /trade/quote para obtener cotización SOL → token. POST /trade/swap para crear la TX de compra. Siempre contra Mainnet Bags (Meteora DBC) independiente de la red BCF.

📊

Analytics

GET /analytics/lifetime-fees · GET /analytics/creators · GET /analytics/claim-stats. Mostrados en el Dashboard con caché de 10 min por mint.

📈

Dexscreener

GET /dexscreener/order/availability para verificar si el token puede listarse. POST /dexscreener/order para crear la orden. Disponible en bags.js, integración UI pendiente.

🏢

Incorporation

Flujo de 3 pasos: POST /incorporate/payment → POST /incorporate (datos legales + founders) → POST /incorporate/start. Accesible post-lanzamiento desde el Dashboard.

Base URL: VITE_BAGS_API_BASE = https://public-api-v2.bags.fm/api/v1
Auth: Header x-api-key: VITE_BAGS_API_KEY en todas las llamadas.
Bags siempre en Mainnet: mainnetConnection en bags.js apunta siempre a api.mainnet-beta.solana.com — independiente de VITE_NETWORK.
